# 任务：将 Gateway 和 LLM 合并为一个可执行程序

当前项目需要分别启动：

- `qq-maid-gateway-rs`
- `qq-maid-llm`

请将它们合并为一个统一可执行程序。以后用户只启动一个程序，即可同时运行 Gateway 和 LLM。

## 实现要求

1. 先检查 workspace、两个项目的 `main.rs` 和实际启动流程。
2. 将 Gateway 和 LLM 的启动逻辑整理为可复用函数。
3. 新增统一入口，同时启动：
   - LLM HTTP 服务
   - QQ Gateway
   - 当前已有的后台任务
4. 暂时保留现有调用方式：
   - Gateway 继续通过本机 HTTP 调用 LLM
   - 本次不要改成进程内直接调用
5. 确保 LLM 已完成监听后，再启动 Gateway。
6. 日志、配置和 `.env` 等全局初始化只执行一次。
7. `Ctrl+C` 或服务异常时，整个程序能够正确退出。
8. 不修改 Todo、RSS、天气、聊天等业务逻辑。
9. 不做无关重构。

## 构建和脚本调整

构建产物可以直接调整，不要求继续保留原来的两个独立二进制。

请搜索并同步修改所有依赖旧二进制名称或启动方式的内容，包括但不限于：

- GitHub Actions
- 构建和打包脚本
- 启动、停止、重启脚本
- 安装脚本
- makefile
- systemd service
- README 和部署文档
- release 打包目录和压缩包内容

所有脚本最终应只启动新的统一程序。

不要出现：

- 脚本仍分别启动 Gateway 和 LLM
- release 包中缺少统一程序
- 停止脚本只能关闭其中一个服务
- 健康检查仍依赖已经删除的旧进程名
- 日志路径、PID 文件或二进制路径未同步更新

## 验收标准

- 构建后只需要一个主要可执行文件。
- 启动该程序后，Gateway 和 LLM 均正常运行。
- Gateway 可以正常调用 `/v1/respond`。
- 原有机器人功能不受影响。
- 现有部署和运维脚本已切换到统一程序。
- 启动、停止、重启和发布脚本可以正常使用。
- `Ctrl+C` 后整个程序完整退出。
- workspace 构建、测试和 Clippy 检查通过。

## 完成后汇报

请说明：

1. 新的统一可执行文件名称和入口。
2. 修改了哪些代码文件。
3. 修改了哪些构建、部署和运维脚本。
4. 删除或废弃了哪些旧启动方式。
5. 如何保证 LLM 先启动完成。
6. 执行了哪些测试及结果。
7. 是否还有未迁移的脚本或已知风险。
